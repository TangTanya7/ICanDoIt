> 视角：资深 C 端需求分析师
> 原则：可跑通完整业务闭环
> 版本：v1.0
> 日期：2026-03-19

---

## 一、业务闭环总览

### 完整转化漏斗

```
[获客] → [激活] → [首日留存] → [周留存] → [月留存] → [目标达成] → [新循环]
  │        │         │            │          │           │            │
  │        │         │            │          │           │            └─ M2 再次触发
  │        │         │            │          │           └─ M8 终极回顾
  │        │         │            │          └─ M8 月度蜕变报告
  │        │         │            └─ M5+M7 锚点提醒+伙伴拉回
  │        │         └─ M6 完成首个任务 + M9 首次进度反馈
  │        └─ M2 设定目标 + M3 愿景激发 + M5 锚点设定
  └─ M1 注册/进入
```

### 每日核心循环（产品的心跳）

```
锚点时刻到达
    │
    ▼
M5 多渠道推送触达 ──→ 用户收到提醒
    │
    ▼
M3 展示未来愿景（3秒）──→ 欲望被激发
    │
    ▼
M6 展示今日任务卡 ──→ 用户点击跳转
    │
    ▼
用户在外部完成学习/练习 ──→ 回到 App
    │
    ▼
M6 确认完成 ──→ M9 连续天数/进度/反馈
    │               │
    ▼               ▼
M7 伙伴反馈    M8 进度更新
    │
    ▼
M10 系统记录行为数据 → 优化明天的任务/资源/难度
```

## 三、逐模块说明

---

### M1：用户账号与引导

**核心业务行为**：让用户以最小阻力进入产品，并完成首次目标设定流程。

| 项目 | 说明 |
|---|---|
| **输入** | 用户的微信授权 / 手机号 |
| **核心行为** | 注册/登录 → 首次引导流（介绍产品价值 → 引导进入 M2 目标设定） |
| **输出** | 用户账号创建完成；用户进入目标设定流程 |
| **关键指标** | 注册转化率；引导流完成率 |

**业务规则**：
- 引导流要在 60 秒内完成，不问多余问题
- 小程序优先用微信一键登录，零输入注册
- 引导流本身就是产品体验的一部分（伙伴语气开场），不是冷冰冰的功能介绍

---

### M2：目标设定与解析

**核心业务行为**：通过对话式交互，将用户的模糊意愿转化为结构化的可执行目标。

| 项目 | 说明 |
|---|---|
| **输入** | 用户的自然语言描述（"我想30天减重10斤"）；或从目标模板库选择 |
| **核心行为** | 伙伴对话引导 → 确认目标品类 → 评估当前水平 → 确认投入时间 → 确认目标周期 |
| **输出** | 结构化目标对象（品类、细分方向、当前水平、目标水平、周期、每日时长）；触发 M3 生成愿景 + M4 生成路径 |
| **关键指标** | 目标设定完成率（开始设定→完成的比例）；从注册到设定完成的平均时长 |

**业务规则**：
- 用户需要回答的问题不超过 3 个（目标是什么？现在什么水平？每天多少时间？）
- 提供热门目标模板，一键选择即可，降低输入成本
- 允许用户后续修改目标参数（周期、每日时长等）
- 一个用户同一时间是否支持多目标？ → **待确认边界问题 1**

---

### M3：欲望引擎

**核心业务行为**：持续生成和维护用户对目标的渴望，覆盖未来愿景、身份代入、进度对比三个子系统。

| 项目 | 说明 |
|---|---|
| **输入** | M2 输出的结构化目标；M8 输出的用户累计进度数据；当前天数/阶段 |
| **核心行为** | ① 首次生成沉浸式未来愿景场景 ② 每日打开时展示愿景（3-5秒） ③ 阶段性更新愿景内容（防疲劳） ④ 生成身份叙事日记（基于真实行为数据） ⑤ 更新身份进化线位置 |
| **输出** | 愿景展示内容（文案+视觉）；身份日记内容；身份进化线当前阶段 |
| **关键指标** | 愿景页停留时长；愿景页到任务页的转化率（看完愿景后开始做任务的比例） |

**业务规则**：
- 愿景内容必须具体、感官化，不能是空洞的鸡汤
- 愿景内容需要定期刷新（至少每周有变化），防止用户跳过
- 身份日记引用用户的真实数据（"你已经学了280个词"），不说空话
- 身份进化线的阶段跃迁是一个仪式性时刻，需配合 M7 伙伴对话

**三个子系统的触发时机**：

| 子系统 | 触发时机 | 频率 |
|---|---|---|
| 未来愿景 | 每次打开 App 时 | 每日 |
| 身份叙事日记 | 达到里程碑（7天/14天/30天...）时 | 阶段性 |
| 身份进化线 | 进入新阶段时 | 阶段性 |

---

### M4：路径规划与资源策展

**核心业务行为**：将结构化目标转化为每日可执行的任务序列，并为每个任务匹配优质互联网资源。

| 项目 | 说明 |
|---|---|
| **输入** | M2 输出的结构化目标；M10 输出的动态调整指令（如果有） |
| **核心行为** | ① AI 生成 N 天的任务路径（分阶段、分任务类型） ② 为每个学习/练习任务匹配互联网资源链接 ③ 为每个任务配置"备选资源"（2-3个） ④ 存储路径供 M6 每日调取 |
| **输出** | 完整的任务路径（每天 2-3 个任务，含类型、描述、资源链接、预计时长）；资源备选列表 |
| **关键指标** | 路径生成成功率；资源链接有效率；用户对推荐资源的好评率 |

**业务规则**：
- 每日任务控制在 2-3 个，总时长不超过用户设定的每日时长
- 每个任务属于三种类型之一：学习（看/听）、练习（动手做）、反馈（记录/自评）
- 任务类型配比随阶段变化：前期学习多、后期练习多
- 每个学习任务默认推荐一个最优资源 + 2个备选 + 允许用户自填链接
- 资源来源优先级：人工策展精选库 > AI 搜索推荐 > 用户自填
- 路径不是一次性全部生成锁死，而是滚动生成（至少提前 7 天），便于 M10 动态调整

---

### M5：锚点触发与触达

**核心业务行为**：在用户日常生活的正确时刻，通过最优渠道，将任务以零阻力的方式送到用户面前。

| 项目 | 说明 |
|---|---|
| **输入** | 用户设定的锚点场景和时间；M10 输出的时间优化建议；今日待完成任务（来自 M6） |
| **核心行为** | ① 用户初始设定 1-3 个锚点（对话式引导） ② 到达锚点时间时，通过最优渠道推送 ③ 推送内容含场景化文案 + 任务摘要 + 深度链接 ④ 用户点击后直达 M6 的任务执行页 ⑤ 基于行为数据持续优化推送时间 |
| **输出** | 推送消息（含深度链接）到达用户的锁屏/微信/短信/小组件 |
| **关键指标** | 推送到达率；推送后 30 分钟内打开率；锚点时间与用户实际活跃时间的吻合度 |

**业务规则**：
- 推送文案必须是场景融合式，不是命令式（"刷牙了？顺便听一段" vs "该学习了"）
- 推送文案由 M7 伙伴引擎生成，保持语气一致
- 深度链接直达当日最该做的那一个任务，不是任务列表首页
- 多渠道优先级（MVP）：微信订阅消息 > App Push > 短信
- 小组件（P1）：桌面常驻展示今日任务 + 连续天数 + 一键开始
- 每天推送次数上限：不超过锚点数量，绝不额外骚扰
- 渐进脱离机制：用户连续 14 天不需要提醒就主动完成时，降低推送频率，并给予"习惯独立"成就

**贴纸系统（MVP 轻量功能）**：
- 用户设定锚点后，系统生成一张可保存/打印的场景贴纸图片（含二维码）
- 二维码扫描后直达当日任务

---

### M6：每日任务中心

**核心业务行为**：承载用户的每日核心交互——看到任务、跳转执行、回来确认。这是产品的「主战场」。

| 项目 | 说明 |
|---|---|
| **输入** | M4 输出的当日任务列表；M9 输出的当前连续天数/进度数据 |
| **核心行为** | ① 展示今日 2-3 个任务卡片（类型标签 + 标题 + 时长 + 跳转按钮） ② 用户点击跳转按钮 → 跳转至外部资源（B站/App/网页等）或内部轻任务 ③ 用户完成后回到 App → 点击"完成"确认 ④ 触发 M9 进度更新 + M7 伙伴评语 ⑤ 全部完成时触发伙伴庆祝 |
| **输出** | 任务完成状态 → 传递给 M8（进度更新）、M9（坚持机制更新）、M10（行为数据） |
| **关键指标** | 日任务完成率（至少 1 个 / 全部完成）；从打开任务到点击跳转的转化率；从跳转到回来确认的回流率 |

**两种任务执行形态**：

| 类型 | 行为 | 示例 |
|---|---|---|
| **跳转型** | 点击 → 跳转外部 → 做完回来确认 | 去B站看教程、去Keep跟练、去Duolingo做课程 |
| **原地型** | 在 App 内直接完成 | 拍照记录饮食、输入今日体重、写一句感受、做一个眼光测试 |

**业务规则**：
- 任务卡片设计遵循"看一眼就知道要做什么、点一下就能开始"原则
- 跳转按钮文案要具体："打开B站看视频"而不是"开始任务"
- 每个任务旁有"换一个"按钮 → 弹出 2-3 个备选资源
- 单个资源可收集用户反馈（这个资源有帮助吗？）
- 弹性机制：每天完成 1 个即算"今日打卡成功"，完成全部有额外奖励
- 未完成的任务不累积到第二天（避免积压压力感）
- 多目标场景：任务按目标分组展示，每个目标用不同颜色/图标区分，用户一眼看清"今天每个目标各要做什么"
- 跳转型任务的完成流程：跳转 → 回来 → 回答 1 道验证题（10秒内可完成）→ 答对算完成 → 答错建议重新去看

### M7：伙伴对话引擎

**核心业务行为**：在所有用户触点上，以一个有温度、有记忆、有性格的角色身份与用户对话，是产品的「灵魂层」。

| 项目 | 说明 |
|---|---|
| **输入** | 当前触点场景（打开App/完成任务/断签回归/里程碑等）；用户全部历史数据（天数、完成量、连续记录、上次活跃时间等）；当前时段/星期 |
| **核心行为** | 根据场景+数据+关系深度，生成自然的、有温度的对话文案 |
| **输出** | 一段伙伴对话文案（用于 App 内各触点展示） |
| **关键指标** | 用户对伙伴文案的满意度（后期可加"这句话暖到你了吗"轻量反馈） |

**伙伴对话覆盖的全部触点**：

| 触点 | 触发条件 | 对话类型 |
|---|---|---|
| 打开 App | 每次打开 | 问候语（根据距上次间隔、时段、天数等变化） |
| 完成单个任务 | 每次完成 | 具体夸奖 + 进度更新 |
| 全部完成 | 当日全勤 | 庆祝 + 身份强化 |
| 未完成时再打开 | 当日有未完成 | 轻度鼓励，不施压 |
| 断签回归 | 中断后首次回来 | 温暖欢迎 + 肯定回归行为 + 降低当日门槛 |
| 里程碑 | 7/14/30/60/90天等 | 深度叙事 + 数据回顾 + 身份确认仪式 |
| 推送消息 | 锚点时间 | 场景融合式提醒 |
| 每周回顾 | 每周末 | 周总结（伙伴帮你回顾语气） |
| 困难检测 | 连续几天低完成率 | 关心 + 主动降低难度提议 |
| 资源反馈差 | 用户给差评 | 道歉 + 换资源 |

**关系深度分级**：

| 阶段 | 天数 | 语气特征 |
|---|---|---|
| 初识 | Day 1-7 | 温和、鼓励、稍正式 |
| 熟悉 | Day 8-30 | 轻松、偶尔俏皮 |
| 老友 | Day 31-60 | 随意、敢开玩笑、更真实 |
| 战友 | Day 61-100 | 深度共鸣、提及共同回忆 |
| 灵魂伙伴 | Day 100+ | 简洁有力，无需多言 |

**反模板原则**：
1. 永远不说"亲爱的用户"——要有称呼的亲近感
2. 永远不说"恭喜你"——说"牛啊！"或更具体的话
3. 永远不说"继续加油"——说"明天见"或具体的下一步
4. 永远引用具体数据——不是"你很棒"，是"你已经学了280个词"
5. 承认不好的状态——"今天不想做也没关系"
6. 用停顿和换行制造呼吸感
7. 关系越深，越像朋友说话

**MVP 实现方式**：
- 文案模板库 + 变量插值（{streak_days}、{last_visit_hours}、{total_tasks} 等）
- 按场景×关系深度×时段建立文案矩阵
- 后期可接入 AI 大模型实现完全动态生成

### M8：进度追踪与回顾

**核心业务行为**：记录用户的全部进度数据，并在 6 个时间维度上生成激励性回顾内容。

| 项目 | 说明 |
|---|---|
| **输入** | M6 输出的每日任务完成记录；M9 输出的连续天数/进度数据变化 |
| **核心行为** | ① 实时更新用户进度数据 ② 在各时间维度触发回顾内容生成 ③ 进度验证（自我对比：Day 1 vs Day N） |
| **输出** | 进度数据面板；各维度回顾卡片/报告；传递给 M3（用于愿景更新和身份日记生成） |
| **关键指标** | 回顾内容的查看率；分享率；回顾后次日留存率 |

**六个时间维度的回顾**：

| 维度 | 触发时机 | 内容形态 | 核心作用 |
|---|---|---|---|
| 每天 | 当日全部完成时 | 今日成就卡（轻量） | 即时满足感 |
| 每周 | 每周日晚/周一早 | 周报卡片 | 阶段成长感 |
| 每两周 | 每14天 | 习惯确认页 | 习惯形成确认点 |
| 每月 | 每月1日/满30天 | 月度蜕变报告（全屏沉浸） | 深度身份蜕变感 |
| 每季度 | 满90天 | 蜕变庆典 | 习惯固化里程碑 |
| 每年 | 满一年/年底 | 年度成长年鉴 | 人生记录感 |

**进度验证方式**：
- 用户的录音/截图/作品的 Day 1 vs Day N 对比（用户主动记录，系统保存）
- 用户自评量表（定期自评 1-10 分）
- 客观数据对比（如阅读速度、词汇量、体重曲线等用户自行录入的数据）

---

### M9：进度与坚持机制

**核心业务行为**：通过简洁的进度追踪和坚持记录，让用户清晰地看到自己在进步。我们是效率工具，不做游戏化。

| 项目 | 说明 |
|---|---|
| **输入** | M6 输出的任务完成事件 |
| **核心行为** | ① 更新连续天数 ② 检查是否触发进度卡片生成 ③ 管理休息日 ④ 触发即时反馈（进度更新 + 伙伴评语） |
| **输出** | 连续天数 + 进度卡片 + 伙伴评语 |
| **关键指标** | 休息日使用率；进度记录回顾频率 |

**坚持机制（仅 3 样）**：

| 机制 | 说明 |
|---|---|
| **连续天数** | 🔥 数字显示，断签重置但伙伴肯定过去的努力 |
| **进度记录** | 里程碑节点自动生成进度卡片（含真实数据），永不消失，可回看可分享 |
| **休息日** | 每周 1 天可标记为休息，不算断签，无需购买 |

**即时反馈设计**：
- 完成一个任务：✅ 打勾动画 + 进度更新（"今日 2/3"）+ 伙伴说一句话
- 全部完成：🎯 "今日全部完成" + 伙伴的具体鼓励
- 核心：反馈靠伙伴那句具体的话，不靠特效

**设计边界：明确不做**——成长树/虚拟宠物、XP、等级系统、虚拟货币、成就徽章、变比率奖励。判断标准：这个功能是帮用户把事做了，还是让用户在"做事"之外多了一件事？只留前者。

---

### M10：动态调整引擎

**核心业务行为**：持续监测用户行为信号，自动调整任务难度、节奏和资源偏好，确保用户始终处于"有挑战但不过难"的最佳区间。

| 项目 | 说明 |
|---|---|
| **输入** | M6 输出的行为数据（完成速度、跳过率、资源评分）；M8 输出的阶段回顾中用户的自评反馈 |
| **核心行为** | ① 分析行为信号 ② 生成调整指令 ③ 传递给 M4 更新后续路径 |
| **输出** | 调整指令 → M4（更新任务难度/节奏/资源偏好） |
| **关键指标** | 调整后任务完成率是否提升；用户反馈的"难度刚好"比例 |

**行为信号与调整动作**：

| 信号 | 判断 | 调整动作 |
|---|---|---|
| 连续 3 天秒完成所有任务 | 太简单 | 通过 M7 提议加难度，确认后提升 |
| 连续 3 天只完成 1 个任务 | 可能太难或太忙 | 通过 M7 关心，主动降低任务量 |
| 某类任务总被跳过 | 对该类型有抗拒 | 降低该类型频率或难度 |
| 某个资源被评差评 | 资源不合适 | 自动替换为备选资源 |
| 周回顾中用户选"太简单" | 用户主动反馈 | 提前进入下一阶段 |
| 周回顾中用户选"有点吃力" | 用户主动反馈 | 增加复习任务，放慢节奏 |
| 用户更偏好视频而非文章 | 资源类型偏好 | 后续优先推荐视频资源 |

**MVP 简版**：先实现"每周回顾问卷（3题）→ 手动调整下周计划"，后期逐步自动化。

## 六、边界问题决策记录

### 决策 1：多目标支持 → 支持，上限 3 个

- **结论**：MVP 支持同时最多 3 个目标，后期可拓展上限
- **设计影响**：
  - M2：目标设定流程支持"添加新目标"入口，已有目标数达上限时提示
  - M4：每个目标独立生成任务路径
  - M5：多个目标的锚点可以不同（如"刷牙时学英语""午饭后看设计教程"）
  - M6：每日任务中心合并展示所有目标的当日任务，按目标分组，每个目标有颜色/图标区分
  - M8：进度追踪按目标独立统计，回顾报告中展示多目标综合进展
  - M9：连续天数全局计算（跨目标），进度记录按目标独立生成

### 决策 2：伙伴人格 → 默认统一人格，后期可拓展

- **结论**：MVP 使用统一的默认人格——热忱、俏皮、真诚、有记忆的老朋友语气（参照前期对话确定的角色定义），后期可拓展为多种可选人格
- **设计影响**：
  - M7：MVP 阶段只需维护一套文案模板库
  - 后期拓展时，文案矩阵按"人格×场景×关系深度"三维扩展

### 决策 3：完成验证 → 轻量验证 + 后期时间检测 + 反作弊惩罚

- **结论**：分阶段实施
  - **MVP**：轻量验证——跳转型任务回来后，附带一个 10 秒内能回答的小问题（选择题），既验证又巩固学习。问题由 M4 在生成任务时一并生成。
  - **V2**：增加时间检测——记录离开时长，任务预计 10 分钟但用户 30 秒就回来，伙伴温和提示"好像有点快？再看看？"，用户可选择"确实看完了"或"重新去看"
  - **反作弊机制**：如果系统检测到用户频繁秒回+答错验证题，触发以下递进惩罚：
    1. 第一次：伙伴温和提醒——"我觉得你好像没太认真看？没关系，再看一遍可能效果更好"
    2. 多次重犯：连续天数不计入——"如果没真的完成，今天就先不算了？"
    3. 持续作弊：伙伴诚实对话——"我注意到最近好几次都跳过了……要不咱们聊聊，是不是任务不太合适？"
  - **核心原则**：惩罚方式保持伙伴语气，不是冷冰冰的"系统检测到异常"，而是朋友温和但诚实的劝诫
- **设计影响**：
  - M4：为每个跳转型任务附带 1 道验证题（选择题，3 个选项）
  - M6：完成确认流程增加验证环节（答题 → 答对→完成 / 答错→建议重新去看）
  - M9：连续天数计入受完成可信度影响
  - M10：累计作弊行为数据，触发递进惩罚
